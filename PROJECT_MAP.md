# Project Map

Живой реестр того, что реализовано в проекте. Цель — чтобы понять "что уже есть и где" можно было по этому файлу, а не обходом всего кода.

**Правило:** создал компонент/сервис/interceptor/роут/включил опцию окружения — добавь строку сюда в том же коммите/PR, где это появилось. Не оставляй это только в коде.

## Структура / конвенции

- Скелет создан `ng new` (Angular 22, standalone, без `NgModule`); тест-раннер — встроенный Vitest-based `@angular/build:unit-test` (дефолт Angular 22, Karma/Jasmine сознательно не используются).
- Папки: `src/app/core/`, `src/app/shared/`, `src/app/features/`, `src/environments/` — назначение и путь-алиасы (`@core/*`, `@shared/*`, `@features/*`, `@env/*`) см. `README.md`.
- `src/environments/environment.ts` (dev) / `environment.prod.ts` — конфиги под алиас `@env/*`, `fileReplacements` в `angular.json` подставляет prod-версию при production-сборке.
- `ApiService` (`stream.Front#3`) проверен юнит-тестом на `HttpTestingController`, а не реальным вызовом к живому backend — на момент реализации у бэка не было ни `/health` (`streamer.API#7`), ни CORS (`streamer.API#5`). Реальная end-to-end проверка — когда обе появятся.
- ESLint (`@angular-eslint`) + Prettier настроены (`eslint.config.js`, `.prettierrc`, `.prettierignore`); `eslint-config-prettier` отключает конфликтующие с Prettier правила форматирования. Команды — см. `README.md`. Git-хуки сознательно не подключены.
- Конвенция loading/error/empty (`stream.Front#9`): loading — `Skeleton`, error — `ErrorMessage` (см. «Компоненты»). Empty-состояние **не** унифицировано намеренно — решается индивидуально по фиче.
- `initializeAuth` (`stream.Front#14`, `src/app/core/initializers/auth.initializer.ts`) — `provideAppInitializer` в `app.config.ts`: вызывает `AuthService.fetchCurrentUser()` до рендера маршрутов, любую ошибку (`401`/сеть) гасит, бутстрап не блокирует. Backend `/auth/me` (`streamer.API#16`) и CORS credentials-режим (`#19`) ещё не реализованы — проверено юнит-тестами на `HttpTestingController`, реальная e2e-проверка отложена (та же ситуация, что у `ApiService`/`AuthService`).

## Компоненты

- `NotificationList` — `src/app/shared/components/notification-list/` — toast-компонент, рендерит очередь `NotificationService.notifications`, крестик → `dismiss()`; подключён один раз в `app.html` (корневой layout)
- `Skeleton` — `src/app/shared/components/skeleton/` — loading-плейсхолдер с shimmer-анимацией, форма/размер под конкретный элемент через inputs `width`/`height`/`radius` (дефолты `100%`/`16px`/`4px`)
- `ErrorMessage` — `src/app/shared/components/error-message/` — единый текст ошибки уровня элемента, дефолт «Ошибка загрузки» (`message` input для редких переопределений)
- `Button` (`stream.Front#39`) — `src/app/shared/components/button/` — перенос приложенного пользователем SVG «Поддержать» (Frame 68, viewBox `0 0 320 51`), 1:1 код из файла, статичный, кроме явно динамических мест: текста, иконки, ширины и типа (без отдельного `variant`-инпута — от той части параметризации отказались по прямому запросу пользователя; но внутренний `uid` всё же есть — не как публичный API, а как техническая необходимость под несколько инстансов на странице, см. ниже). Текст — `text()` (`input.required<string>()`); typography — Figma-компонент «Button» (Montserrat Bold 18/24, letter-spacing 8%), цвет — вертикальный градиент (зависит от `type()`, см. ниже) через `background-clip: text`, задаётся через `[style.backgroundImage]` (НЕ `[style.background]` — шорткат `background`, даже заданный инлайново, сбрасывает `background-clip` обратно на `border-box`, даже если `background-clip: text` задан отдельно с более низким приоритетом в `button.scss`, — текст заливался сплошным прямоугольником вместо обтекания по форме букв, поймано визуально на `/kit`; `background-image` — самостоятельный longhand, эту проблему обходит). Иконка — ромбовидная подложка+монетка, вынесены в статичный ассет `public/icons/button-support-icon.svg`, подставляются через content projection (`<ng-content select="[icon]" />`). Иконка+текст — единая flex-группа `.button__content`, отцентрованная относительно **всей кнопки целиком** (не относительно исходных координат макета), зазор между ними — ровно `8px` (`gap`); если иконка не спроецирована, `.button__icon:empty { display: none }` убирает слот из раскладки чисто через CSS — тогда текст один по центру. Никакой JS-логики под это не нужно (более ранняя версия на `hasIcon()`/`viewChild`/`afterRenderEffect` заменена на чистый CSS flexbox).
  `width()` (`input<number>()`, px) — «фиксированные пиксели», 9-slice-подобное растягивание: скруглённые/острые концы, центральный зазор под гем и декоративные кольца — фиксированного размера (в исходных unit), растягивается только плоская середина. Реализовано как 3 слоя, каждый со **своими** границами тип/скруглений (их вершины контура не совпадают между слоями — каждый вписан на свою глубину): «glow» (`filter0_i`, границы `28.1421`/`291.858`), «рамка/обводка» (`filter5_d`, границы `30.0527`/`289.947` — на ~2 unit уже, чем у glow), оба разбиты на 5 `clip-path`-частей (острый край / растягиваемая половина / зазор 20px / растягиваемая половина / острый край); и «тело» (заливка+2 градиента, `filter1_i`, границы `30.0523`/`289.948`), 3 части (острый край / единый растягиваемый блок / острый край). Зазор под гем (`150`–`170`) и его сдвиг, а также сдвиг правого острого конца (`translate(extra 0)`) — общие формулы у glow и рамки (`gapTransform()`/`rightTipTransform()`), т.к. не зависят от того, где именно у слоя начинается скругление; только сам якорь/scale растягиваемых половин у каждого слоя свой (`glowMidLeftTransform()`/`glowMidRightTransform()` vs `frameMidLeftTransform()`/`frameMidRightTransform()`) — переиспользование ОДНИХ И ТЕХ ЖЕ clip-path/transform между glow и рамкой было ошибкой (рамка чуть уже, граница «съедала» кусок скругления на стыке блоков), эту границу развели по прямому запросу пользователя. Растяжение — `[attr.transform]`, `scale` анкорится в исходной вершине контура (`translate(anchor+shift) scale(sx) translate(-anchor)`), картинка не искажается ни у одной фиксированной части. px/unit (48/51) — константа независимо от ширины, поэтому острые концы всегда 1:1 с оригиналом. **Кольца (`filter2_d`, внутри `mask0_2821_998`) — по прямому запросу пользователя вообще не масштабируются** (растяжение сплющило бы окружности в эллипсы): остаются исходным одним `path` без `clip-path`-разбиения, вся маскированная группа только сдвигается — `centerShiftTransform()` (`translate(extra/2 0)`), та же формула, что и у гема (`filter3_d`), чтобы оба остались по центру новой ширины. Угловые диамант-блики (маленький path с 2 подконтурами у остриёв в исходнике) вынесены из `mask0_2821_998` и разбиты на 2 отдельных `path`: левый вообще без transform (как острый левый конец, никогда не двигается), правый — `rightTipTransform()` (та же формула `translate(extra 0)`, что и у правого острого конца) — иначе оба блика двигались бы вместе с `centerShiftTransform()` (`extra/2`) и отставали от реально расходящихся остриёв (которые расходятся на всю `extra`).
  `width()` кроме числа (px, «фиксированные пиксели» выше) принимает `'parent'` и `'content'` (тип `ButtonWidthMode`, экспортирован из `button.ts`) — оба реального размера в px не знают заранее (зависят от раскладки браузера/родителя/шрифта), поэтому измеряются `ResizeObserver` в конструкторе и пишутся в обычные `signal()` (не `computed()` — это легитимный случай синхронизации с внешним, не-Angular источником, не анти-паттерн «effect дублирует состояние»), которые уже участвуют в общем `widthPx()` наравне с числовым режимом — вся 9-slice-геометрия ниже (glow/рамка/тело/кольца/гем/блики) от режима не зависит вообще, работает одинаково для всех трёх. `'parent'`: `.button` получает CSS `width: 100%` (не инлайновый px — иначе кнопка «застревала» бы на последнем измерении вместо реальной реакции на ресайз родителя/окна) через отдельный `widthStyle()`, а `ResizeObserver` на самом `.button` (`#buttonEl`) только измеряет фактический результат для внутренней геометрии SVG. `'content'`: ширина — иконка+`8px`-зазор+текст (естественная, НЕ растянутая ширина `.button__content-inner` — обычный flex-блок, в отличие от родителя `.button__content`, у которого `position: absolute; inset: 0` принудительно растягивает бокс на всю кнопку и сделал бы измерение бессмысленным) + `CONTENT_PADDING_PX` (`50`) слева и справа; одно и то же измерение реагирует и на смену `text()`, и на появление/исчезновение иконки (пустая `:empty` иконка схлопывается в `0` через CSS, `ResizeObserver` это тоже видит). `ResizeObserver` недоступен в jsdom (юнит-тесты) — код это учитывает (`typeof ResizeObserver === 'undefined'` guard, нужен только ради тестовой среды, не ради прод-браузеров), поэтому под `'parent'`/`'content'` в тестах измеренные сигналы остаются на начальном значении (`DEFAULT_WIDTH_PX`/`0`) — реальная раскладка (что кнопка правда тянется на всю ширину контейнера, что `'content'` действительно даёт разную ширину под разный текст) проверена вручную через Playwright на живом dev-сервере, не автотестом.
  `type()` (`input<ButtonType>('primary')`, `ButtonType = 'primary' | 'secondary'`, экспортирован из `button.ts`) — по прямому запросу пользователя (приложил `Frame 68_2.svg`, `viewBox 0 0 176 51`) меняет **только цвета**, геометрия/9-slice/размеры общие для обоих типов (проверено построчным сравнением обоих исходных SVG — вся остальная разметка идентична). Из SVG взяты 4 значения (`computed()`, привязаны к `[attr.fill]`/`[attr.stop-color]`/`[style.backgroundImage]`): `glowFill()` (`#F4E9AE`/`#8383F3`, все 5 частей `filter0_i`), `bodyBaseFill()` (`#EEC68C`/`#3F3FAF`, первый из 3 fill-слоёв `filter1_i` во всех 3 блоках), `bodyTintStart()`/`bodyTintEnd()` (стопы `paint1_radial_2821_998`: `#9B8474`→`#D5AF71`/`#26267B`→`#7171D5`) и `textGradient()` (CSS-строка целиком, `#77531C`→`#653908`/`#7C7CFA`→`#DCDCFC`). Рамка/блики/гем в исходнике буквально те же `F7ECB2`/`FFF9DB`/`F8ECB2`, что и у primary — по прямому запросу пользователя ("тоже перекрасить") перекрашены и они, но это **подобранный** (не из исходника) лавандовый тон `#DCDCFC`/`#F0F0FF`: `sparkleFill()`/`gemFill()` — обычный `[attr.fill]`, отработал штатно. **Рамка (`frameStrokeUrl()`) — особый случай**: реактивная мутация `stop-color` у уже существующего `paint4_radial_2821_998` (тот же приём, что сработал для `bodyTintStart/End` через `fill="url(#paint1_radial)"`) для рамки **не сработала** — DOM-атрибут обновлялся верно, а закешированный рендер `filter5_d` — нет (тот же класс Chromium-бага, что и со статичным `width` региона фильтра, см. ниже, но новый нюанс: специфично для `stroke="url(#...)"`, для `fill="url(#...)"` тот же паттерн работает исправно — проверено пиксельно на живом рендере, зафиксировано в памяти `project_svg_filter_dynamic_attr_bug`). Фикс — тот же принцип, что и с `width`: не мутировать существующий ресурс, а завести **два полностью статичных** `<radialGradient>` (`paint4_radial_2821_998` / `paint4_radial_2821_998_secondary`) и реактивно переключать, на какой из них ссылается `[attr.stroke]`.
  **Шов между clip-path-частями:** соседние части стыкуются встык математически точно (нулевой зазор проверен аналитически), но два независимо anti-aliased `path` не гарантированно ложатся на один под-пиксель — на глаз был виден тонкий (~1px) тёмный шов на границах. Зафиксировано **только у рамки** (`filter5_d` — острые концы и её собственный `clip-frame-mid-center_2821_998`, не общий с glow, продлены на `0.2` unit внутрь соседних растягиваемых частей — безопасно, т.к. у рамки drop-shadow фильтр, не блюрящий саму заливку). У glow/тела (`filter0_i`/`filter1_i`) такой же фикс **намеренно не сделан** — у обоих inner-shadow blur (сам paint размывается), и малейшее перекрытие расползается blur'ом в заметное яркое пятно (проверено на глаз при попытке); их шов остаётся как есть — на практике куда менее заметен, сама заливка и так мягкая от blur. Продлевать РАСТЯГИВАЕМУЮ (scaled) часть в сторону соседа нельзя нигде — кусок контура рядом с границей ещё не плоский (кривая скругления заканчивается точно на границе), scale исказил бы его.
  **`id`/`url(#...)` — уникальны на инстанс (`uid`, было известным ограничением, теперь исправлено).** Изначально все `id`/`url(#...)` у фильтров/градиентов/масок/clip-path внутри главного SVG были захардкожены как в исходнике (`_2821_998`) — при единственном инстансе на странице это работало, но при нескольких `<app-button>` (ровно ситуация `/kit`) браузер резолвит `url(#id)` (и `document.getElementById`) в **первый** элемент с таким `id` в ДОКУМЕНТЕ целиком, а не в тот, что лежит в собственном `<svg>` конкретного инстанса — было незаметно, пока все инстансы красились одинаково (primary), и стало реальным видимым багом с появлением `type()`: секондари-кнопка своим DOM (внутри своего `<svg>`) показывала верный синий `paint1_radial`, а РИСОВАЛА primary-жёлтый — потому что `fill="url(#paint1_radial_2821_998)"` резолвился в градиент самой первой primary-кнопки на странице (диагностировано сравнением пиксельного рендера с raw-SVG вне Angular + `document.getElementById` в живом DOM). Исправлено: `uid` — инкрементный счётчик на класс (`let nextButtonUid = 0` в module scope, `protected readonly uid = \`btn${nextButtonUid++}\`` в конструкторе), добавляется суффиксом ко ВСЕМ id/url(#...) в шаблоне (`[attr.id]="'xxx_2821_998-' + uid"` и т.п. — именно `[attr.x]`, не `{{}}`-интерполяция прямо в статичном атрибуте: для гифенизированных SVG-атрибутов типа `clip-path`, а также `in2`/`result` на `<feBlend>`, Angular пытается скомпилировать интерполяцию как property binding и падает с NG8002 «not a known property», т.к. они не в его схеме известных DOM IDL-свойств — `[attr.x]` этого обходит). Внутренние имена `result=`/`in=`/`in2=` на filter-примитивах (`effect1_dropShadow_2821_998`, `effect1_innerShadow_2821_998`) `uid` НЕ получили — они по SVG-спеке видны только внутри своего `<filter>`, не глобальный `id`, конфликтовать не могут. Тест на регрессию — `button.spec.ts` рендерит 2 инстанса (primary+secondary) одновременно в `document.body` и проверяет через `document.getElementById`, что `url(#...)` каждого резолвится в его собственный ресурс.

  **Известные ограничения (сознательные):** (1) ширина `filter0_i`/`filter1_i`/`filter5_d` регионов (`x`/`width`) — заведомо большое статичное число (`5000`), **не** `[attr.width]` с вычисляемым сигналом: эмпирически найдено, что Chromium не пересчитывает закешированный рендер SVG-фильтра, если его bounding-box атрибуты меняются через императивный `setAttribute` (именно так Angular применяет attribute binding) уже после вставки элемента в DOM — растянутая середина кнопки просто не рисовалась за пределами исходного (маленького) региона фильтра, при этом сами `clip-path`/`transform` атрибуты применялись и пересчитывались корректно. Статичное большое значение из разметки работает без проблем (детали — memory `project_svg_filter_dynamic_attr_bug`). (2) мелкий хвостик-указатель в середине body-слоя (`filter1_i`) не защищён отдельным зазором — при сильном растяжении/сжатии слегка масштабируется вместе с остальной серединой (не считается существенным дефектом). Демо — роут `/kit` (`KitPage`): без иконки, с иконкой, и `width()` на 4 размерах (150/301/500 с иконкой/700px).
- `SectionTitle` (`stream.Front#37`) — `src/app/shared/components/section-title/` — заголовок секции с декоративным подчёркиванием, перенос приложенного SVG (Group 471, viewBox `0 0 210 53`). Из исходника взята только нижняя декоративная часть (2 затухающие линии + шеврон по центру, `filter1_d_2906_2094`) — верхняя часть (текст, запечённый как `path` с тенью, `filter0_d_2906_2094`) удалена целиком, текст живой: `text()` (`input.required<string>()`), рендерится обычным `<h2>` (не `<h1>` — компонент используется по несколько раз на одной странице, `<h1>`×N сломал бы document outline; `<h2>` — тот же уровень, что уже использует `kit-page.html` для своих подзаголовков). Тень — CSS `text-shadow` (не портированный SVG-фильтр — проще и не тянет за собой класс Chromium-багов с filter-регионами, см. `Button` выше). Typography — миксин `mixins.h1` (см. Design tokens), первый компонент, использующий миксин, а не россыпь токенов инлайн. `id`/`url(#...)` внутри SVG — с суффиксом `-uid` с самого начала (не как исправление задним числом, как у `Button`) — тот же incremental-counter приём (`let nextSectionTitleUid = 0`, `protected readonly uid = \`sectiontitle${nextSectionTitleUid++}\``), тест на 2 одновременных инстанса в `document.body` есть сразу.
  **Известное ограничение (сознательное, по прямому запросу пользователя — «перенеси как есть, дальше будем работать»):** подчёркивание зафиксировано на 210px, как в исходном SVG, НЕ подстраивается под длину текста — при коротком тексте подчёркивание нависает за него, при длинном текст перетекает за подчёркивание; адаптивная ширина — отдельная будущая задача. Демо на `/kit` намеренно показывает оба случая (короткий/длинный текст рядом) — см. секцию «подчёркивание зафиксировано на 210px».
  **Открытый вопрос по типографике (не обрабатывается в этой задаче):** из трёх реальных мест использования («Расписание», «Топ донатеров», «Соц. сети» — `docs/figma/main1.json`/`main2.json`, умбрелла-репо) только «Расписание»/«Топ донатеров» соответствуют стилю «H1» (Montserrat Bold 28px), «Соц. сети» в Figma-экспорте — Montserrat 22px/500, несовпадение. `SectionTitle` реализует только стиль «H1»; при подключении к `SocialLinksBlock` (`stream.Front#32`) типографику для «Соц. сети» нужно будет сверить отдельно, не считать совпадающей автоматически.
- `Shell` (`stream.Front#19`) — `src/app/shared/components/shell/` — header/nav по мокапу `main-nav`, контент через `<ng-content>`; подключён один раз в `app.html` вокруг `<router-outlet>`, поэтому переиспользуется на всех страницах автоматически. Nav-пункты (Главная/Новости/Турниры) и кнопка входа (статичный гостевой вид, без `AuthService`) — временные: реальное содержимое `main-nav` (лого, точные пункты меню) не выгружено из Figma API из-за rate limit (см. `figma.md`), требует доуточнения повторным `get_figma_data`, когда лимит снимется. `.shell__header` фон — `transparent` (не сплошной цвет), изменено в `stream.Front#28`: по макету главной страницы меню лежит поверх полноэкранной карусели `MainCarousel`; действует глобально на все страницы, т.к. `Shell` один на весь сайт.
- `MainCarousel` (`stream.Front#28`) — `src/app/features/main/components/main-carousel/` — специфичный для главной страницы компонент карусели (не абстрагирован под общий переиспользуемый слайдер), `position: fixed; inset: 0` на весь viewport (Full HD и кратные — другие разрешения не проверялись). Ровно 2 слайда (`activeIndex` signal, `next()`/`prev()`/`goTo()` с зацикливанием). Картинка слайда — `imageUrl0`/`imageUrl1` (URL, источник hero-изображений этой задачей не определён — на превью тестовые файлы `public/images/main-carousel/`) + `imagePosition0`/`imagePosition1` (`'left' | 'center' | 'right'`, на всю высоту, `object-fit: contain`). Контент — 3 именованных слота контент-проекции по фактической разметке макета: `[mainCarouselSlide0BottomRight]` (Расписание), `[mainCarouselSlide0BottomLeft]` (Соц. сети), `[mainCarouselSlide1BottomLeft]` (Топ донатеров) — сами виджеты не реализованы (задачи `#30`/`#31`/`#32`), здесь заглушки. При смене слайда картинка и контент уезжают в сторону своего же позиционирования (`translateX` + `opacity`, без общего fade всего слайда — во избежание визуальной вспышки при наложении). Стрелки навигации — точный SVG из макета (`public/icons/main-carousel-arrow.svg`, правая стрелка зеркалируется через `transform: scaleX(-1)`). Автопрокрутка — 10 секунд на слайд, зациклена (`setInterval` 50мс тик, `progress` signal 0..1); таймлайн внизу — сегмент на слайд, активный заполняется в реальном времени по `progress`, пройденные — полностью залиты; ручная навигация (стрелки/`goTo`) сбрасывает отсчёт. Проверено юнит-тестами (навигация, автопрокрутка/сброс таймера на fake timers, классы позиции картинки, рендер спроецированного контента через host-компонент).

## Сервисы

- `ApiService` — `src/app/core/services/` — методы: `get()`, `post()`, `put()`, `patch()`, `delete()` (принимают опциональный `{ withCredentials }`); собирает URL из `environment.apiUrl` + путь, единая точка `catchError` для generic сетевых ошибок (без завязки на доменные ошибки)
- `NotificationService` — `src/app/core/services/` — методы: `show(message, type, durationMs?)` (типы `success`/`error`/`info`; `durationMs: null` — persistent-уведомление, не скрывается само), `dismiss(id)`, `dismissAll()` — очередь на `signal`, поддерживает несколько одновременных уведомлений
- `AuthService` (`stream.Front#4`) — `src/app/core/services/` — `currentUser` (readonly signal `CurrentUser | null`), `isAuthenticated` (computed); методы `login(login, password)`, `loginWithGoogle(googleIdToken)`, `fetchCurrentUser()` — `POST/GET` через `ApiService` с `withCredentials: true` (JWT в `httpOnly`-cookie, фронт токен не хранит); `logout()` — `POST /auth/logout`, сбрасывает `currentUser` в `null` через `finalize()` независимо от результата запроса. Backend-эндпоинты (`streamer.API#16-18`) и CORS credentials-режим (`#19`) ещё не реализованы — проверено юнит-тестами на `HttpTestingController`, реальная e2e-проверка отложена.

- `ProfileService`/`SettingsService` (`stream.Front#21`) — `src/app/core/services/` — `getProfile()`/`updateProfile(dto)` (`GET`/`PATCH /profile`), `getSettings()`/`updateSettings(dto)` (`GET`/`PATCH /settings`), через `ApiService`. Модели (`Profile`, `Settings`, `UpdateProfileDto`, `UpdateSettingsDto`) списаны с реального контракта backend (`streamer.API#21`/`#23`, оба уже реализованы и закрыты — DTO/контроллеры прочитаны напрямую из `backend/src/profile/`, `backend/src/settings/`, не угаданы). Несмотря на готовый backend, реальная e2e-проверка не сделана — локально не поднята MySQL/`.env` для полного стека (это уже за рамками frontend-задачи); проверено юнит-тестами на `HttpTestingController` с точными путями/методами/телами по факту прочитанного кода.
- `ImageUrlService` (`stream.Front#22`) — `src/app/core/services/` — `resolve(path)` строит полный URL картинки из относительного пути backend-хранилища (`${environment.apiUrl}${path}`); отдельная опция окружения под base URL картинок не заведена — backend раздаёт `/uploads/*` статикой с того же origin, что и API (`useStaticAssets`, см. `backend/src/main.ts`), поэтому `environment.apiUrl` уже покрывает разницу dev/prod.
- `GoogleAuthService` (`stream.Front#15`) — `src/app/core/services/` — грузит Google Identity Services JS SDK (`https://accounts.google.com/gsi/client`) динамическим `<script>`-тегом (не инлайновый скрипт в `index.html`), кэширует промис загрузки. `signIn()` инициализирует SDK с `environment.googleClientId`, дожидается ID-токена через `google.accounts.id.prompt()` и передаёт его в `AuthService.loginWithGoogle()`. Отмена/закрытие попапа и недоступность SDK не роняют приложение — отдаются как ошибка Observable вызывающему коду. Без UI — кнопка/попап входа появятся вместе с auth-модалкой (макапы ещё не пришли); проверено юнит-тестами с замоканным `window.google` и `loadScript()`.

## Interceptors / Guards

- `withCredentialsInterceptor` (`stream.Front#14`) — `src/app/core/interceptors/` — форсит `withCredentials: true` на всех HTTP-запросах приложения (не только через `AuthService`/`ApiService`), чтобы httpOnly auth-cookie уходила глобально; подключён в `app.config.ts` через `provideHttpClient(withInterceptors([...]))`
- `authInterceptor` (`stream.Front#5`) — `src/app/core/interceptors/` — на `401` вызывает `AuthService.logout()` и показывает уведомление через `NotificationService` (без редиректа — окно логина/регистрации реализовано как модалка, без отдельного роута). Исключение — запросы к `/auth/*`: `401` от `/auth/me` (гость, см. `#14`) и `/auth/login`/`/auth/google` (неверные креды) не триггерят logout, а `/auth/logout` явно исключён из собственной же логики, иначе получился бы бесконечный цикл. Ошибка не глотается — пробрасывается дальше вызывающему коду. Backend-эндпоинты и CORS credentials-режим (`streamer.API#16-19`) ещё не реализованы — проверено юнит-тестами на `HttpTestingController`.
- `authGuard` (`stream.Front#20`) — `src/app/core/guards/` — `CanActivateFn`, проверяет только факт авторизации (`AuthService.isAuthenticated()`), без ролей (RBAC — отдельная задача). Неавторизованных редиректит на `/` (не на `/login` — окно логина модалка, без роута). Ложный редирект до завершения `fetchCurrentUser()` при старте исключён архитектурно — `provideAppInitializer` (`#14`) блокирует первую навигацию роутера до готовности `currentUser`, поэтому сам guard не нуждается в асинхронном ожидании. Пока не подключён ни к одному реальному роуту — защищённых маршрутов ещё нет (появятся вместе с будущими feature-задачами).

## Роуты

- `/main` (`stream.Front#28`) — `src/app/features/main/pages/main-page/` — **временная превью-страница**, не финальный Main (тот — отдельная задача `#33`, заменит/дополнит текущий роут). Рендерит `MainCarousel` с заглушками вместо ещё не реализованных Schedule/Donators/Social виджетов — для ручной проверки компонента.
- `/kit` (`stream.Front#39`) — `src/app/features/kit/pages/kit-page/` — служебная страница-каталог UI-kit компонентов (не часть пользовательского сайта), сейчас показывает `Button`: без иконки, с иконкой, `width()` числом на 4 размерах (150/301/500 с иконкой/700px), `width()="parent"` в обёртках заданной ширины (400px/700px, `.kit-page__parent-demo` с dashed-outline — чтобы визуально было видно, что кнопка реально дотягивается до края), `width()="content"` на разных текстах/с иконкой, и `type()="secondary"` (без иконки/с иконкой/с `width()`); и `SectionTitle` (`stream.Front#37`): 3 реальных заголовка блока Main, плюс короткий/длинный текст рядом — специально показывает нестыковку фиксированной ширины подчёркивания с длиной текста — для ручной сверки при вёрстке.

## Design tokens / переменные

- `src/styles/_variables.scss` — 18 цветовых переменных (`$color-<hex>`, литеральные имена, без выдуманных ролей типа `$primary`) и типографика (`$font-family-montserrat`/`$font-family-nunito-sans`, `$font-size-14..28`, `$font-weight-400..700`, `$line-height-36`, `$letter-spacing-8`), выведены из `docs/figma/*.json` (`designTokens`, умбрелла-репо). Подключаются через `@use 'variables' as vars;` — `includePaths` настроен в `angular.json`.
- `src/styles/_mixins.scss` (`stream.Front#37`) — миксины, названные по буквальному имени Figma text-стиля (не выдуманная роль), та же конвенция именования, что у `_variables.scss`. Пока один: `h1` (`font-family`/`size`/`weight` + новые `$line-height-36`/`$letter-spacing-8`) — Figma text-стиль «H1» (Montserrat Bold 28/36, letter-spacing 8%), первый компонент с именованным многосвойственным text-стилем (раньше, у `Button`, типографика собиралась россыпью токенов инлайн в самом компоненте — там это осталось, миксин не ретроактивен). Подключается через `@use 'mixins' as mixins;`, применяется как `@include mixins.h1;`.
- `src/styles/_reset.scss` — global reset (box-sizing, обнуление margin/list/link-стилей), применяется один раз через `src/styles.scss`.
